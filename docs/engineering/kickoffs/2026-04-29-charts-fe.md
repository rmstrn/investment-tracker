## 1. Why this slice

The static showcase at `/design-system.html` and the runtime showcase at `/design#charts` already demonstrate all 11 chart variants visually (PRs #76 + #77 + #78). However, those are static prose-and-mock-data demonstrations — they cannot consume a real AI agent payload, cannot enforce Lane-A structural guardrails at runtime, and cannot be reused in chat / dashboard / positions surfaces.

This slice ports those 11 demonstrations into 11 production React components that:

1. **Render directly from a typed `ChartPayload` discriminated union** — no per-call configuration; the chart kind drives the renderer choice via a single `switch (payload.kind)` branch.
2. **Validate Lane-A structural constraints at parse time, not runtime** — Risk Flags 1 / 2 / 3 from CHARTS_SPEC §5.2 are absent from the Zod schema entirely (not optional+policed), so the parser rejects forbidden payloads before the component is even mounted. Backend cannot accidentally ship a violating chart.
3. **Live-switch theme via `var()` resolution** — zero JS-runtime cost, no remount on theme change. Per chart blueprint architectural decision 3.
4. **Tree-shake correctly** — T1 charts (Line / Area / Bar / Donut / Sparkline / Calendar / Treemap) are eager exports from `packages/ui/charts`; T2 (StackedBar / Scatter / Waterfall) and T3 (Candlestick) are lazy from `packages/ui/charts/lazy`. Routes that never render a chart pay 0 kb. The chat route pays only the T1 chunk (~95 kb gz total per blueprint §Bundle size).

After this slice, the AI agent can emit a `ChartEnvelope` JSON, the frontend parses it once at the API-client boundary, and a single `<ChartRenderer payload={parsed.data}>` (or 11 per-kind named imports — see §6) renders the right chart with palette + a11y + reduced-motion + empty/error/skeleton states baked in.

This is the last MVP-blocker for chart-bearing AI answers in chat.
