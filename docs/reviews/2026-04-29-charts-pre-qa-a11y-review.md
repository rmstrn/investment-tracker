# Charts pre-QA accessibility review

**Date:** 2026-04-29
**Reviewer:** accessibility-architect
**Scope:** static-source audit of `packages/ui/src/charts/*.tsx` (10 chart components + 7 `_shared/` primitives)
**Branch:** `chore/plugin-architecture-2026-04-29`
**Standard:** WCAG 2.2 Level AA + product spec `docs/design/CHARTS_SPEC.md` v1.1 §3 / §7
**Read-only:** no code changes; no runtime axe-core (QA owns runtime).

---

## 1. Verdict

**FIX-BEFORE-MERGE.**

Two structural failures block AA conformance: (a) the visible focus indicator is suppressed via inline `outline: 'none'` on every chart container while `tabIndex={0}` makes them keyboard-reachable — direct violation of **WCAG 2.4.7 Focus Visible (AA)** and **WCAG 2.4.11 Focus Appearance (AA, new in 2.2)**; (b) the `useChartKeyboardNav` hook is **defined but never imported** by any of the 10 chart components — the kickoff §7 acceptance criterion («arrow-keys cycle data points; Esc blurs») is unmet. Both are reachable from production today via the showcase route; AT users who tab into a chart land in a silent dead-zone with no visible focus and no arrow navigation.

Beyond these two, six HIGH findings cluster around contrast on the Treemap text-on-fill combinations, a `role="alert"` misuse in `ChartError`, and the Sparkline's mandatory focusability contradicting its spec status as «glance-only, not the primary read».

The 10 components do nail the easy wins: `role="img"` + `aria-label` + `aria-describedby` are wired on every container; `<ChartDataTable>` provides a real `<table>` data-equivalent with stable `useId()`-generated ids; `prefers-reduced-motion: reduce` is honoured uniformly via `isAnimationActive={!prefersReducedMotion}` on every Recharts root; `aria-hidden="true"` is correctly applied to decorative SVG markers in Calendar; Lane-A captions (drift / Treemap FINRA / Waterfall mechanical-decomposition) are present.

## 3. Findings — by severity

### CRITICAL

#### CRITICAL-1 — Visible focus indicator suppressed on all 10 charts (WCAG 2.4.7 AA + 2.4.11 AA)

Every chart container sets `style={{ outline: 'none', ... }}` while also carrying `tabIndex={0}`, making the chart keyboard-reachable but unable to display any default browser focus ring. No replacement `:focus-visible` rule, Tailwind utility, or `box-shadow` outline is wired up at the package or app level.

Files (all at the chart container `<div>`):
- `packages/ui/src/charts/LineChart.tsx:115`
- `packages/ui/src/charts/AreaChart.tsx:58`
- `packages/ui/src/charts/BarChart.tsx:87`
- `packages/ui/src/charts/DonutChart.tsx:59`
- `packages/ui/src/charts/Sparkline.tsx:62`
- `packages/ui/src/charts/Calendar.tsx:148`
- `packages/ui/src/charts/Treemap.tsx:114`
- `packages/ui/src/charts/StackedBar.tsx:55`
- `packages/ui/src/charts/Waterfall.tsx:148`
- `packages/ui/src/charts/Candlestick.tsx:87`

Spec contract violated: `CHARTS_SPEC §3.8 line 334` — «On focus, draws the system focus ring: `outline: 2px solid var(--accent); outline-offset: 2px`». Also `§7.2 line 1843`. WCAG 2.2 SC 2.4.11 (Focus Appearance) requires the indicator to have a 2px-min thickness and 3:1 contrast against adjacent colour — currently zero indicator.

**Recommended fix:** drop `outline: 'none'` from inline style and add a `:focus-visible` rule to the package's CSS (or apply via Tailwind className `focus-visible:outline focus-visible:outline-2 focus-visible:outline-offset-2 focus-visible:[outline-color:var(--accent)]` at the container). The inline `outline: 'none'` was likely added to suppress the Recharts SVG focus halo — if that's the rationale, scope it to a child selector, not the focusable host.

#### CRITICAL-2 — `useChartKeyboardNav` never imported; arrow-key navigation absent on all 10 charts

The hook at `packages/ui/src/charts/_shared/useChartKeyboardNav.ts:13-58` is exported but no chart component imports it. Grepping `useChartKeyboardNav` across `packages/ui/src/charts/*.tsx` returns zero hits. Result: a keyboard user who Tabs onto a chart cannot move through data points; arrow keys, Home/End, and Esc do nothing.

Spec contract violated: `CHARTS_SPEC §3.8 line 335` — «Arrow keys cycle through data points». Kickoff §7 line 216 — «`tabIndex={0}` on container; arrow-keys cycle data points; Esc blurs». Acceptance criterion unmet.

This compounds CRITICAL-1: even with a visible focus ring, focusing the container yields no further navigation. AT users get a silent landing.

**Recommended fix:** wire the hook into each chart with an `activeIndex` `useState` that drives Recharts' `<Tooltip active={true} payload={data[activeIndex]}>` — the spec implementation note (line 1850) describes the pattern explicitly. Sparkline can opt out (no tooltip), but the other 9 should consume it.

### HIGH

#### HIGH-1 — Treemap small text fails AA contrast on negative + neutral tiles

`packages/ui/src/charts/Treemap.tsx:54-78` renders ticker label + percentage at `fontSize: 10` and `fontSize: 11` using `fill="var(--color-text-inverse, #fff)"`. Tile fills are decided by `colorForTreemapChange()`:

- `treemapNegative` = `var(--chart-series-2)` — light `#A04A3D`. White on `#A04A3D` ≈ **3.99:1 — FAILS AA body 4.5:1** (passes UI 3:1 and AA Large at 18px+, but tile text is 10–11px).
- `treemapNeutral` = `var(--chart-series-5)` — light `#67756F`. White on `#67756F` ≈ **3.81:1 — FAILS AA body**.

Worst case: a payload with `dailyChangeBasis` absent forces every tile to neutral grey (per `colorForTreemapChange` falling through), giving a wholly non-compliant treemap. WCAG 1.4.3 (Contrast Minimum) AA fail.

**Recommended fix:** raise tile text to ≥14px-bold (qualifies as «large text» at 3:1) OR switch text colour to `var(--color-text-primary)` and rely on the dark tile being light enough — verify per-token. Alternative: darken `series-2` light by `-0.06 L` for treemap-only use.

#### HIGH-2 — `ChartError` uses `role="alert"` for static error placeholder

`packages/ui/src/charts/_shared/ChartError.tsx:30` sets `role="alert"`, which is an assertive live region — screen readers interrupt the user's current task to announce the error. Appropriate for a runtime failure that appears mid-interaction, not for the chart's initial-render error state when a payload fails Zod parsing — that case the SR user is browsing into the error, not being interrupted by it.

WCAG 4.1.3 (Status Messages) AA — alerts should be reserved for time-sensitive interruptive feedback. Repeated chart errors on a dashboard would create rapid-fire announcements.

**Recommended fix:** default to `role="status"` (polite); accept an optional `urgent: boolean` prop for the rare interrupt case. Or `role="region" aria-labelledby={headlineId}`.

#### HIGH-3 — `ChartError`'s disabled fallback button is meaningless to AT

`packages/ui/src/charts/_shared/ChartError.tsx:52-60`: when `?debug=1` is absent, a `<button disabled aria-label="Show payload">Show payload</button>` is rendered. AT announces «Show payload, button, dimmed» with no explanation of why it's disabled. WCAG 3.3.1 (Error Identification) — affordance is unclear.

**Recommended fix:** when not in debug mode, render nothing OR render `<p className="text-xs text-text-tertiary">Payload hidden — append <code>?debug=1</code> to the URL to inspect.</p>`. The disabled button adds keyboard noise (still tab-stoppable on some browsers) without value.

#### HIGH-4 — Sparkline forced focusable contradicts spec § 4.5

`packages/ui/src/charts/Sparkline.tsx:54-58` always sets `role="img" tabIndex={0} aria-describedby`. CHARTS_SPEC §4.5 line 681 explicitly says: «sparklines do NOT get focusable role=img — they sit alongside a number that already conveys the value. The `<svg>` is `aria-hidden="true"`. The accompanying number IS the data; the sparkline is a glance.»

Result: in dense surfaces (KPI tiles, table cells with adjacent number), a Sparkline now adds an extra Tab stop and an extra SR announcement of the same value. Standalone-only exception (line 683) is not implemented.

**Recommended fix:** add a prop `standalone?: boolean` (default `false`). When `false`, render with `aria-hidden="true"` and no `tabIndex`. When `true`, render the current focusable + transcript form. Update consumers (KPI tiles) to pass `standalone={false}` explicitly.

#### HIGH-5 — Calendar dividend status `received` vs `scheduled` colour-only on visual surface

`packages/ui/src/charts/Calendar.tsx:111-128` distinguishes `received` (filled circle, accent) from `scheduled` (filled circle, series-4) by colour-fill alone. Both share circular geometry and both are `aria-hidden="true"` (correctly — they are decorative chips). The `title` attribute carries status text but is mouse-only.

The CHARTS_SPEC §7.3 line 1861 acknowledges this gap and accepts «hover tooltip and screen-reader transcript both surface status as text» as the mitigation. The transcript IS present in `ChartDataTable.tsx:172-181` («Dividend ({e.status})»). So **AT users have a path** via the visually-hidden table.

What's missing: a sighted colour-blind user (deuteranopia / protanopia) on the visual grid sees two filled circles of similar luminance and cannot distinguish status without mouse-hovering each. Per CHARTS_SPEC §7.3 line 1859 («Calendar … Sub-status within `dividend` (received vs scheduled) is the only purely-color distinction; mitigation: hover tooltip and screen-reader transcript») — this is documented as the chosen trade-off.

**Recommended fix (defensive, not strictly required for AA):** introduce a shape variation — e.g. solid-circle for `received`, half-filled for `scheduled` — to remove the residual colour-only encoding. WCAG 1.4.1 (Use of Color) AA — currently relies on documented spec mitigation; tightening would be safer.

#### HIGH-6 — Calendar grid has no per-cell focus / no per-event semantics

`packages/ui/src/charts/Calendar.tsx:139-213` renders a 7×6 cell grid where each cell is a styled card (border, padding, hover-affordance look) but is non-focusable, non-clickable, and unsemantic. A keyboard user has no way to land on a specific day; a SR user reading the visual grid hears the chart label only and must rely on the visually-hidden `<ChartDataTable>` to access events. The grid cells visually look interactive (CSS gives them card affordances at `min-h-[64px] rounded-md border bg-background-elevated p-1.5`).

Strictly, this is acceptable per CHARTS_SPEC §3.8 (the chart is one focusable unit + transcript). But the visual styling implies interactivity, which violates WCAG 3.2.4 (Consistent Identification) when these cards look like the elevated cards used elsewhere as buttons.

**Recommended fix:** either flatten cell visual styling so they don't read as cards (drop border + bg-elevated; keep grid structure), OR make individual cells focusable (`tabIndex={-1}` + roving tabindex pattern with the existing `useChartKeyboardNav`). The latter is the spec direction once nav is wired (CRITICAL-2).

### MEDIUM

#### MEDIUM-1 — `ChartEmpty` `role="status"` without explicit `aria-live`

`packages/ui/src/charts/_shared/ChartEmpty.tsx:34`. ARIA spec says `role="status"` has implicit `aria-live="polite"`, but several SR engines (older NVDA, JAWS scripts) still need explicit `aria-live="polite"` for reliable announcement. Add for cross-AT robustness.

#### MEDIUM-2 — `ChartSkeleton` lacks `aria-busy`

`packages/ui/src/charts/_shared/ChartSkeleton.tsx:38-42`. `role="status" aria-label="${kind} chart loading"` is good. Add `aria-busy="true"` so AT users know to wait — clean skeleton state is one of the most common SR pain points.

#### MEDIUM-3 — Calendar `EventMarker` `title` is mouse-only

`packages/ui/src/charts/Calendar.tsx:97-127`. `title` attribute is keyboard-inaccessible and inconsistently announced across SR engines. Move semantic info to `aria-label` on a focusable wrapper, or rely entirely on the ChartDataTable transcript (current fallback). Currently the transcript covers it, so this is a defence-in-depth note.

#### MEDIUM-4 — DonutChart legend not responsive per spec § 4.4

`packages/ui/src/charts/DonutChart.tsx:89-94` hardcodes `align="right" verticalAlign="middle" layout="vertical"`. CHARTS_SPEC §4.4 line 632 calls for legend right at ≥768px and bottom at <768px. On narrow viewports the legend overflows the donut wrap. Not strictly an a11y fail, but reflows poorly at 320px (WCAG 1.4.10 Reflow AA).

#### MEDIUM-5 — `aria-label` falls back to potentially empty `meta.title`

Every chart at the container `aria-label={payload.meta.alt ?? payload.meta.title}`. If `meta.title === ''` (Zod allows empty strings if not `.min(1)`-constrained — verify in shared-types schema), the chart ships with `aria-label=""` which is equivalent to no label on `role="img"` — WCAG 1.1.1 fail. Defensive fallback recommended: `payload.meta.alt ?? payload.meta.title || \`${payload.kind} chart\``.

#### MEDIUM-6 — BarChart drift detection English-substring-only

`packages/ui/src/charts/BarChart.tsx:49-51`. `subtitle.toLowerCase().includes('drift')` won't match localized payloads. Today the platform is English-only so no AA impact, but the drift caption is regulatory-grade prose — guard against future localization with an explicit payload flag (`payload.subtype === 'drift'` per spec §4.3).

### LOW

#### LOW-1 — `ChartDataTable` caption is `<p>` sibling, not `<caption>`

`packages/ui/src/charts/_shared/ChartDataTable.tsx:38`. Spec example at `CHARTS_SPEC §7.4 line 1884` uses `<caption>` inside the `<table>`. SR engines associate `<caption>` more reliably with the table than a sibling `<p>`. Move the caption inside each `renderXxxTable` function.

#### LOW-2 — Sparkline ChartDataTable falls back to `<p>` summary

`packages/ui/src/charts/_shared/ChartDataTable.tsx:148-157`. Spec consistency would prefer a 2-row table even for sparklines. Cosmetic; SR users still get the data.

#### LOW-3 — Treemap ChartDataTable rows render raw numbers

`packages/ui/src/charts/_shared/ChartDataTable.tsx:198-220`. `weightPct`, `valueAbs`, `dailyChangePct` rendered as bare numbers (`12.4`, not `12.4%`; `184000`, not `$184,000`). SR reads literal numbers without unit, which is technically WCAG 1.3.1 (Info and Relationships) — units carry meaning. Pipe through `formatValue` with `'percent'` / `'currency-compact'`.

#### LOW-4 — Inline `outline: 'none'` is also on Sparkline (consistency note)

`packages/ui/src/charts/Sparkline.tsx:62`. If Sparkline becomes opt-in focusable per HIGH-4, the same focus-ring mandate applies; ensure the standalone branch gets `:focus-visible` styling.

## 5. Recommendation

**FIX-BEFORE-MERGE.**

The two CRITICAL findings (suppressed focus ring + dead keyboard-nav hook) are AA fails that the kickoff §7 acceptance criteria explicitly require to be passing before merge — they are not theoretical regressions, they are unmet contract clauses. Both are small, surgical fixes (≤ 1 line per chart for the focus ring; ~10 lines per chart to wire the hook) and do not require schema or spec changes.

HIGH-1 (Treemap text contrast) is also a structural fail visible in the showcase route today.

HIGH-2 / HIGH-3 / HIGH-4 are localised to shared primitives and a single chart; ~30 minutes of work each.

HIGH-5 is documented as the chosen trade-off in CHARTS_SPEC §7.3 — accept-as-is OK, but flag in PR description.

HIGH-6 is a stylistic warning that compounds CRITICAL-2; resolves naturally once keyboard nav is wired to a roving-tabindex pattern.

After CRITICAL fixes + HIGH-1 / HIGH-2 / HIGH-3 / HIGH-4, the chart subsystem reaches WCAG 2.2 AA on the static-source dimensions a code-only audit can confirm. Runtime axe-core verification (QA kickoff) can then run with high signal-to-noise.

Suggested gate: BLOCK merge until CRITICAL-1, CRITICAL-2, HIGH-1, HIGH-2, HIGH-3, HIGH-4 are addressed; APPROVE-WITH-NITS for the rest (file as TDs against the QA slice).
